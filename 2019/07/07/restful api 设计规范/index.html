restful api 设计规范

Veröffentlicht am 2019-07-07 | in 规范
学习下restful风格

URI

URI规范

  • 不用大写
  • 使用中划线 - 而不使用下划线 _
  • 参数列表要加密
  • URI中名词标识资源集合，且需要使用复数形式

资源集合和单资源

集合：

单个：

避免层级过深

/表示层级，用于按实体关联关系进行对象导航，一般跟进id

尽量使用查询参数代替路径的实体导航

对组合资源的访问

组合实体的生命周期依赖于父实体，无法独立存在，不对应表也没有id，通常是对数据库表中某些列进行抽象得到的。例如User-Address Address即country/city这类字段的简单抽象，不独立于User中，需要通过User索引Address

Request

HTTP方法

通过白哦准的HTTP方法对资源进行CRUD

GET:查询

POST:创建单个资源，一般向资源集合发起

PUT:更新单个资源的全部信息，客户端需要提供完整的更新后的资源。与之对应的又PATCH,PATCH只负责更新一部分信息，客户端只提供需要更新的字段即可，PUT/PATCH一般向单个资源发起

DELTE:删除

安全性和幂等性

安全性：不改变资源的状态，即只读

幂等性：执行一次和执行多次，对资源状态的改变是等价的

XXX 安全性 幂等性
GET 有 有
POST 无 无
PUT 无 有
DELETE 无 有

安全性和幂等性均无法保证反复的请求能拿到相同response，第一次删除可以返回200，第二次则可以返回404

复杂查询

XXX 示例 备注
过滤条件 ?type=1&age=22 允许冗余 /zoos/1 /zoos?id=1
排序 ?sort=age,desc
投影 ?whitelist=id,name,email
分页 ?limit=10&offset=3

BookMarker

经常使用的复杂的查询需要标签化以降低维护成本

Format

  • Content-Type:application/json

  • Content-Type:application/x-www-form-urlencoded

  • Content-Type:multipart/form-data; boundary=—-RANDOM_jDMUxq4Ot5(带文件上传的表单)

Content Negotiation

资源有多种表示的方式，例如json\xml\pdf\excel等，客户端可以指定期望的格式

  • http header Accept

  • url文件后缀

response的body只是数据，不再做多余的包装，例如：

处理格式

XXX 具体格式
GET 单个对象或集合
POST 新增成功的对象
PUT/PATCH 更新成功的对象
DELETE 空

json的时间用长整，客户端再按需解析

不传null字段

错误处理

  • 发生了错误不能给2xx响应，客户端会缓存成功的http请求
  • 正确设置http状态码，无需自定义
  • response body提供错误代码和描述文本

第三点实现

java一般使用异常表示restful api的错误，api会抛出两大类异常：业务和非业务异常，业务异常由自己的代码抛出，表示用例的前置条件不满足，业务规则冲突，例如：参数校验失败、权限校验失败等，非业务异常为非预期问题，通常是因为类库和框架抛出，例如数据库连接异常、空指针、除0等。

业务异常需要提供：HTTP的响应状态码和文本描述，在Controller可使用统一的异常拦截器进行拦截，设置HTTP响应状态码，非业务异常统一使用500，错误码需要提供异常类名。

还需要错误描述，用于指定错误文本，对非业务类异常，可以统一文案“服务器错误，请稍后再试”，服务器端需要提供改行为的开关。

状态码 场景
400 bad request 参数校验错误
401 unauthorized 未验证的用户，例如未登录
403 forbidden 无权限
404 not found 资源不存在
500 internal server error 非业务类异常
503 service unavaliable 由容器抛出，服务器端不能手动抛出此异常

服务型资源

额外的服务无法使用URI映射，例如：

  • 关键字搜索
  • 计算距离
  • 向用户推送消息

这些服务即为资源，计算的结果就是资源的presentation,按服务属性选选择合适的方法

异步任务

对于耗时的异步任务，服务器接收参数后，应返回创建成功的任务资源且需要包含任务执行状态，客户端可以轮训该任务获得执行进度

若任务执行状态信息过多，则可吧执行状态抽象为组合资源

版本

URI失效

对失效的API，返回404 not found 或 410 gone；对迁移的API，返回 301 重定向。
